lens.

Lightweight in-cluster observability for LLM and AI/ML serving on Kubernetes.

Inspect inference pods, tail accelerator-bound workloads, and kubectl exec into a model server — without leaving the browser. One Bun process, in-cluster service-account auth.

Live demo → GitHub

What it does

◆LLM endpoint discovery

Scans every cluster Service on its declared TCP ports for an OpenAI-compatible /v1/models response. Auto-detects vLLM, TGI, llama.cpp, Ollama, sglang, Triton — model list, probe latency, inferred runtime. No tokens spent.

◆Pod & workload browser

Namespaces, pods, deployments, services, configmaps, events — built to surface the inference layer and per-tenant quotas.

◆In-browser kubectl exec

A real shell on any pod via xterm.js over WebSocket. Inspect model weights or attach to a running vLLM process.

◆Resource pressure

Per-namespace CPU/memory pressure, surfaced for multi-tenant serving where one heavy workload can starve neighbours.

◆Security panel

Failed-auth events from collected log streams — useful when fronting LLM endpoints with auth_request.

◆Single-binary scope

Vue 3 + Vite frontend, Bun TypeScript backend. No Electron download, no separate auth proxy.

Why this exists

The big Kubernetes dashboards are general-purpose and heavy. AI/ML serving has different debugging needs: inference pods are large and slow to start, and you need to know exactly which one is in ContainerCreating versus actually serving traffic.

kubectl exec is the fastest way to confirm a vLLM/TGI process is healthy without exporting /metrics for every diagnostic. Multi-tenant serving leans on ResourceQuotas and KEDA — you want pressure visible per namespace, not just per node.

Run locally

# requires Bun and access to a Kubernetes cluster (kind / k3d / k0s)
bun install
bun run dev       # frontend
bun run server    # backend

Architecture

Vue 3 SPA ──HTTPS/WS──▶ Bun server (TS) ──K8s API──▶ control plane xterm.js ◀────────── • REST proxy • exec WS bridge • SA-token rotation

The server reads its bearer token from the standard service-account mount and proxies authenticated calls to the API server — no external secrets store needed.
