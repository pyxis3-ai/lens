01What it does

◆LLM endpoint discovery

Probes every cluster Service on its declared TCP ports for an OpenAI-compatible /v1/models response. Auto-detects vLLM, TGI, llama.cpp, Ollama, sglang, Triton — model list, probe latency, inferred runtime. No tokens spent.

◆Pod & workload browser

Pods, deployments, statefulsets, services, configmaps, secrets, PVCs, ingresses, jobs, cronjobs, events, certificates — a fast keyboard-driven table for the whole cluster.

◆In-browser kubectl exec

A real shell on any container via xterm.js over WebSocket. Inspect model weights, tokenizer state, or attach to a running vLLM process — no terminal, no kubeconfig.

◆Resource pressure

Live per-node and per-pod CPU/memory from the metrics API, surfaced for multi-tenant serving where one heavy workload can starve its neighbours.

◆Alerts & thresholds

Editable CPU/memory/disk/swap/load thresholds, OOMKill and crash-loop detection, plus cert-expiry warnings — pushed live over WebSocket with optional webhook.

◆Security panel

fail2ban jails, SSH brute-force, nginx WAF/ModSecurity blocks, and Authelia auth events from collected log streams — useful when fronting LLM endpoints.

02A closer look

Discovery

Every endpoint, auto-detected.

lens enumerates every Service, ranks candidates by container-image heuristics, and probes /v1/models in batches with a 1.5 s timeout. Runtime is read from data[0].owned_by, then the Server header, then the image. Only the model-list endpoint is hit — never a completion — so discovery costs zero tokens.

Exec

A real shell, in the browser.

Open /bin/sh on any container over the Kubernetes exec WebSocket, bridged through xterm.js with full TTY and live resize. The fastest way to confirm a vLLM/TGI process is healthy without exporting /metrics for every diagnostic.

Workloads

The whole cluster, one keystroke away.

Tab through pods, deployments, statefulsets, daemonsets, services, configmaps, secrets, PVCs, ingresses, jobs and cronjobs. ⌘K spotlight, j/k navigation, inline describe, scale and rolling-restart — built for keyboard speed.

Pressure

Per-namespace, not just per-node.

Multi-tenant LLM serving leans on ResourceQuotas and KEDA. lens surfaces CPU/memory pressure where it matters — per namespace and per pod — so you can see which tenant is starving the others before it pages you.

Security

Failed-auth at a glance.

Live counts and top offenders across fail2ban jails, SSH brute-force attempts, nginx WAF blocks, and Authelia authorization events — the signals you want when an inference endpoint is exposed to the internet.

03Why lens

LLM-firstThe headline view is your inference endpoints — not a generic pod list bolted onto a model cluster.
Single-binary scopeOne Bun process serves the API, the WebSocket bridge, and the SPA. No Electron download, no separate auth proxy.
In-cluster by designReads its bearer token from the service-account mount and proxies the API server. No kubeconfig to export, no external secrets store.
Read-only by defaultMutations (delete, scale, restart, exec) are scoped to the RBAC you grant the ServiceAccount — nothing more.
No tokens spentEndpoint discovery only hits /v1/models. Your GPUs keep serving; lens never sends a completion.
LightBuilt for the operational reality of large, slow-starting inference pods — quick to load, quick to read.

04Under the hood

Vue 3 SPA ──HTTPS / WS──▶ Bun server (TS) ──K8s API──▶ control plane xterm.js ◀─────────────── • REST proxy • exec WS bridge • SA-token rotation

The server reads its bearer token from the standard service-account mount and proxies authenticated calls to the API server. Metrics come from /proc and the metrics API; nginx, fail2ban and Authelia signals from mounted log streams. All in one container.
